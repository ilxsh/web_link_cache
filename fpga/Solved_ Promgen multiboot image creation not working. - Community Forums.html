Promgen multiboot image creation not working.

Jump to solution

I have three bitstreams plus a datafile (elf image) that I am trying to combine into a SPI flash mcs file.

Promgen it appears does not follow either my explict address assignments for the images nor does it actually

place the images at the address it reports if I let it do the assignments via the -n option.  I am trying to program the

image named "gold", "tx", "rx" in that order followed by the elf data file. What I see is that the tx image ends up at address 0x0 where the gold image should be, the gold image is not included (nothing valid is stored in the second location),  and that the rx image is correctly saved at the third location.  The Elf image also appears to be programmed correctly (at least most of the time). 

 

Can anyone confirm that multiboot mcs file creation is working (or not) in 14.4 or 14.6.  or can you find the error in the steps I use.

 

Thanks

Bill Anderson

Re: Promgen multiboot image creation not working.

Jump to solution


 

Turns out the main problem I was having was due to the fact that I upgraded to a 512Mbit SPI flash chip that I had configured Bitgen properties with  to use 32 bit mode addresses. 

  If you do this you need to shift the IPROG WBSTAR address 8 bits before writing

it to the ICAPE fifo. See below. 

 

From Xilinx I got the following info. 

------------------------------------------------------

There is one caveat actually when using 32 bit addressing mode and ICAP.  You need to do the same thing that Bitgen does when 32 bit addressing mode is enabled and cut off the lower 8 bits of the address to turn it basically back into a 24 bit address.  So take my example of 0x1000000 and cutting off the lower 8 bits turns it into 0x10000.  You cannot have a non zero starting address like 0x10000FF for instance because the lower 8 bits are cut off and ignored.

 

If you run through the GUI flow and enable 32 bit addressing mode there is a INFO message that is printed to the console during Bitgen that tells you about this.

 

INFO:Bitgen:369 - To allow for correct fallback functionality the lower 8 bits

   of the value "0x1000000" specified by the "-g next_config_addr" option are

   ignored.  MultiBoot and fallback only work with 256 byte boundaries when "-g

   SPI_32bit_addr:Yes" is used to enable 32 bit SPI addressing.


 

Regards

Bill
